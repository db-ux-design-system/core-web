# Generate Figma Screen (DB UX)

> **MANDATORY:** If you are about to write ANY Figma node code (createInstance, appendChild,
> setBoundVariable, createFrame, etc.) WITHOUT having first authored a Composition Plan JSON
> and made the runtime available — EITHER via the store-once loader (`assets/bootstrap/render.js`,
> PREFERRED) OR by pasting `assets/db-figma-runtime.min.js` verbatim (fallback) — → STOP
> immediately. Read this skill from the top. There is NO shortcut. Every screen goes through
> `renderPlan`.
>
> **OUTPUT IS ALWAYS A FIGMA FRAME — NEVER HTML, CSS, JSX or any workspace file.**
> If you are about to create or edit a file in the workspace: STOP.
> The only permitted output is a rendered frame in the Figma file via `use_figma` + `renderPlan`.

Turns a prompt into a DB UX-compliant Figma screen. The agent authors a declarative
**Composition Plan (JSON)** and hands it to the hardened **render runtime**
(`assets/db-figma-runtime.js`). The agent never hand-writes imperative Figma node code —
that is what makes the result model-independent.

## Pre-Conditions

1. `context/figma-generation.md` is loaded (the binding design rules).
2. A Figma design file with **write** access is connected via the host Figma MCP
   (the render tool, e.g. `use_figma`).
3. `{figma_target}` is a Figma URL with a `node-id`. If not → STOP and ask.
4. The `db-ux` MCP server is connected (for live component/token/icon verification).

## Workflow

### Phase 1 — Resolve the target & scope

1. Extract the `fileKey` and `node-id` from `{figma_target}`. If no `node-id` → STOP, ask.
2. If the screen needs a component that only exists in Core Lab (Concept) and
   `{concept_components}` is not enabled → ask once, then wait.

### Phase 2 — Compose the Plan (per page type → JSON)

Compose from the page type's captured catalogs (see `context/figma-generation.md` → Part A —
Composition), NOT ad-hoc:

1. **Detect the page type** — `dashboard` or `landingpage` (see Part A → "Page type
   detection"). If neither clearly fits → STOP, describe the gap, and propose a new page-type
   catalog for human review (never silently invent structure).
2. **Load that page type's catalog** under `assets/registries/<pageType>/`:
    - `sections.json` — the CONTENT→SECTION selection table (`whenToUse` intents). This drives
      the layout.
    - `template.json` — which sections are required/optional/forbidden + their order + rules.
    - `block-patterns.json` — section-level patterns that a section resolves to.
    - `blocks.json` — the palette of atomic blocks (registry-valid plan fragments).
    - `example.json` — a DENSITY/STYLE reference only (spacing, zebra, heading hierarchy). Do
      NOT copy its section sequence.
3. **Compose content-driven (NOT by copying the example):**
    1. Split the prompt into content groups (the distinct things to show).
    2. Map each group to a section via `sections.json` → `whenToUse` (pick the intent that fits
       the group's shape; record the id + a one-line reason). Never invent — an unmatched group
       → STOP, flag `requiresHumanReview: true`.
    3. Order the chosen sections per `template.json` (positions + rules; avoid a mono-layout
       when content is heterogeneous).
    4. Fill each section from its `pattern`/`itemBlock`: expand `{ "$ref": ... }`, fill EVERY
       `<placeholder>` with real content, resolve actions via the Action Hierarchy, drop an
       optional node ONLY when no content applies (never an empty Select/Icon/Button). Do not
       restructure a captured fragment. If a needed block is a `plan: null` stub → STOP/report.
4. Assemble into ONE Composition Plan following the PLAN SCHEMA at the bottom of
   `assets/db-figma-runtime.js`. Every screen:
    - starts with a `Header` (DB logo + app name) as the first child;
    - uses titled content Sections (`title` + optional `description`) — never a bare grid;
    - starts on `color.background.canvas` (level-1), then alternates the zebra;
    - uses ONE uniform gap (default `lg`) for all direct children of a content block — no
      nested tighter `2xs`/`xs` sub-groups; a Card wrapping a block matches it via `spacing`
      (block `lg` → Card `spacing: "large"`); list/status rows use `spread` + trailing `hugWidth`;
    - applies the page-wide density (`spacing`) and `contentWidth` from `template.json`.
5. Reference ONLY registered names: components + props (`components.json`), color/spacing/radius
   tokens and text styles (`tokens.json`), section/pattern/block ids (the page type's
   `sections.json` / `block-patterns.json` / `blocks.json`).

### Phase 3 — Validate the Plan

1. Resolve every component + variant against `components.json`, and every token + text style
   against `tokens.json`. If anything is missing → STOP and report the exact gap.
2. Run the **composition + visual lint** (`context/figma-generation.md` → Validation / Linting):
   each content group maps to a section via `sections.json` → `whenToUse`; sections ordered per
   `template.json`; no mono-layout for heterogeneous content; every block came from that page type's
   `blocks.json` / `block-patterns.json` (no invented blocks); action hierarchy (≤1 brand
   button per page, equal items share one action kind, no nested interactive elements in a
   clickable card); weight/color agreement; Gestalt grouping; density/width; zebra + header.
   Flag any not-yet-captured block or new page type as `requiresHumanReview: true`.
3. (Optional, recommended) cross-check names live via `db-ux/list_components`,
   `get_component_props`, `list_icons`, `get_design_tokens`.

### Phase 4 — Render via the runtime

There are two ways to get the runtime into the sandbox. **Prefer the store-once path** — it
keeps every render call tiny (~6–7 KB) and avoids the output-budget truncation that a ~33 KB
one-shot paste can hit on smaller models. Fall back to the single verbatim paste only if the
document cannot be bootstrapped.

#### 4a. Store-once (PREFERRED)

The runtime is stored ONCE per Figma file in `figma.root` shared plugin data (which DOES
persist across `use_figma` calls, unlike `globalThis`), then every render/edit only pastes a
tiny loader + the Composition Plan. All snippets are generated by the build into
`assets/bootstrap/` and are byte-generated from the FULL runtime, so they never drift.

1. **Check** whether this file is already bootstrapped: paste `assets/bootstrap/check.js` and
   read the returned meta. Compare its `sha` to `version_sha` in `assets/bootstrap/manifest.json`.
2. **Bootstrap only if needed** (no meta, or `sha` differs → stale/old runtime): paste each
   `assets/bootstrap/store-<i>.js` verbatim in its own `use_figma` call (each ~7 KB, well
   under budget; each returns the stored length so you can integrity-check), THEN paste
   `assets/bootstrap/store-meta.js` LAST (writing meta last makes a partial bootstrap
   detectable). This is a rare one-time cost per file.
3. **Render** in ONE call: paste `assets/bootstrap/render.js` (the loader — it reads the
   chunks back, rehydrates the runtime via `new Function`, and exposes `renderPlan` /
   `applyEdits`), then append:
    - `const PLAN = { screen, targetNodeId, layout, variables };`
    - `const res = await renderPlan(PLAN);`
    - `return JSON.stringify(res.audit);`

#### 4b. Single verbatim paste (FALLBACK)

1. In ONE render call: paste the entire contents of **`assets/db-figma-runtime.min.js`**
   verbatim (the size-optimized build of `db-figma-runtime.js` — the commented source exceeds
   the `use_figma` 50 000-char limit, so the ~33 KB min build is what you paste). `globalThis`
   does NOT persist between calls, so runtime + plan must go in ONE call — never split;
   then define `const PLAN = ...`, `const res = await renderPlan(PLAN);`,
   `return JSON.stringify(res.audit);`.

#### Both paths

- Do NOT hand-roll `createInstance` / `appendChild` / `setBoundVariable`. The runtime already
  handles every Figma API gotcha (see the red-flags table).
- **Editing the runtime**: change `db-figma-runtime.js` (the readable source), then regenerate
  the minified build AND the bootstrap snippets with `node assets/build-runtime.cjs` before
  rendering. Never edit `db-figma-runtime.min.js` or `assets/bootstrap/*` by hand. Because the
  build re-hashes the runtime, any change bumps `version_sha`, so files bootstrapped with the
  old runtime are detected as stale by step 4a.1 and transparently re-bootstrapped.

### Phase 5 — Audit, fix, report

1. Read `res.audit`. If `valid === false`, fix the PLAN and re-render (max 3×).
2. Report: used components (+ source library + maturity), bound variables, used icons
   (functional vs illustrative), unresolved gaps, final validation result.

## Iterating on an existing screen (DO NOT regenerate for small changes)

> **RULE — intent words always refer to the ALREADY-RENDERED screen.** Once a screen exists
> for the target page, follow-up requests using words like **"change", "adjust", "add",
> "append", "extend", "remove", "rename", "swap", "fix", "ändern", "anpassen", "ergänzen",
> "hinzufügen", "entfernen"** ALWAYS mean an in-place edit of that existing frame via
> `applyEdits` — NEVER a fresh `renderPlan`. A full `renderPlan` is ONLY for the first-time
> creation or a genuine structural rebuild (new page type, reordered sections, different
> template). When unsure, first inspect the target page with `figma/get_metadata`; if a frame
> already exists there → `applyEdits`. Doing a full re-render for an additive/edit request is
> a MISTAKE: it wastes tokens (resends the whole plan), loses node ids + manual tweaks, and
> leaves a duplicate frame.

`renderPlan` is for creating a screen the FIRST time. For a **follow-up change to a screen
that is already rendered** (relabel, recolor, hide an element, swap a variant, add/remove a
block), do NOT re-author the whole plan and re-render — that destroys the frame, loses node
ids, and throws away any manual designer tweaks. Instead **patch in place with `applyEdits`**:

1. Paste the **COMPACT edit bundle `assets/db-figma-edit.min.js`** verbatim (~8 KB, not the
   ~33 KB full runtime — this is the biggest cost lever per iteration; the sandbox has no
   `fetch` (can't load the runtime from a URL) and no persistent globalThis between calls, so
   a paste is still required, just a much smaller one).
   Then author a SMALL edit spec (the diff, not the whole screen):
    ```js
    const res = await applyEdits({
    	screen: "Bahnhofsinformationen – Startseite", // exact frame name (or rootId: "12:34")
    	targetNodeId: "700:4960", // optional: page to switch to
    	edits: [
    		{
    			op: "setText",
    			find: "Live-Abfahrten",
    			value: "Echtzeit-Abfahrten"
    		},
    		{ op: "hideNavItem", label: "Startseite" },
    		{ op: "setVisible", name: "Primary Action", visible: false },
    		{
    			op: "setVariant",
    			find: "Reiseinfos in Echtzeit",
    			axis: "As",
    			value: "h3"
    		},
    		{ op: "setContainerGap", anchorText: "Schnellzugriff", gap: "xs" },
    		{
    			op: "setSectionFill",
    			anchorText: "Schnellzugriff",
    			token: "color.background.elevated"
    		},
    		{ op: "remove", find: "Fundservice" },
    		{
    			op: "appendLike",
    			find: "Servicezeiten",
    			node: {
    				/* a full plan node, e.g. a Card */
    			}
    		}
    	]
    });
    return JSON.stringify(res);
    ```
2. Selectors are text-anchored (`find` / `anchorText`, add `mode:"contains"`) or by node
   `name`. Every edit returns `{ op, ok, error? }`; `applyEdits` also re-runs the audit.
3. Edit ops: `setText`, `setVisible`, `hideNavItem`, `setVariant`, `setContainerGap`,
   `setSectionFill`, `setTextFill`, `remove`, `appendLike` (renders a new sibling block via
   the same runtime — so inserts stay registry-valid).
4. Only fall back to a full `renderPlan` when the change is a large structural restructure
   (new page type, reordered sections, different template). For everything else, `applyEdits`.
5. Two bundles exist: **`db-figma-edit.min.js`** (~8 KB, `applyEdits` only — use for
   iterations) and **`db-figma-runtime.min.js`** (~33 KB, full `renderPlan` + `applyEdits` —
   use for first-time creation, or when an edit uses `appendLike`, which needs `renderNode`).
   Both are built by `node assets/build-runtime.cjs` from their `.js` sources.

## Output Checklist

- [ ] `{figma_target}` had a node-id (else stopped and asked)
- [ ] Page type detected (dashboard|landingpage) → its `sections.json` + `template.json` + `blocks.json` (+ `block-patterns.json`) loaded
- [ ] Content split into groups; each mapped to a section via `sections.json` → `whenToUse`; sections ordered per `template.json`; no mono-layout for heterogeneous content; `example.json` used only as density/style reference
- [ ] Action hierarchy respected (≤1 brand button/page; equal items share one action; clickable cards have no nested control)
- [ ] Plan validated — every component/variant resolves in `components.json`; every token/text style in `tokens.json`; every block id in the page type's catalog
- [ ] Rendered via `renderPlan` (runtime pasted verbatim) — no hand-written node code
- [ ] `res.audit.valid === true`
- [ ] Header is the first child; first section is level-1; zebra alternates
- [ ] Header: NO `Startseite`/`Home` nav item (logo = home); meta-nav + action icons OFF unless a real action exists (no empty ✕ placeholders)
- [ ] Heading hierarchy: section `title` = h2, inner content headings are h3/h4/h5 — never two same-size headlines in one section
- [ ] No single-block sections except hero / closing CTA / one media-text row; card content is TOP-aligned (only the trailing action sits bottom-right)
- [ ] Every content Section has a title; each content block uses ONE uniform gap (Card `spacing` matches); images use a ratio (1:1/3:4/16:9); icons use the `Icon` component with a specific `name` (e.g. `arrow_right`, resolved via `registries/icons.json`) — not the bare placeholder; multi-page nav items are named
- [ ] Report produced (page type, blocks/patterns used, components, tokens, icons, gaps, candidates, validation)
